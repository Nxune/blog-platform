## 三、开发经验教训

### 好的做法（继续）

1. **先规划再开发**：Architect 产出设计文档 → Leader 审查 → Coder 实现
2. **Git 实时推送**：每步 commit + push，中文标题
3. **安全先行**：Sec-Engineer 在代码完成后立即审查，发现4个漏洞（含1个CRITICAL）
4. **测试覆盖**：98 单元测试确保回滚不破坏现有功能
5. **简约 UI**：用户喜欢，不要过度设计

### 需要改进的

1. **Agent 响应延迟**：Architect 和部分成员响应慢，Leader 不要越权替他们干活
2. **不要半成品提交**：Phase 2 的 Like 功能是 Schema未迁移就先写前端的典型案例
3. **权限配置前置**：项目开始时就配好 `.claude/settings.json`，避免中途反复授权
4. **不要删提交**：用 revert 而非 rebase 删除提交，保持历史可追溯
5. **团队规模**：9人团队对个人项目过大，4-5人足够（Senior-Coder + QA + Security + DevOps）

### 技术选型建议

- **不要换 ORM**：Prisma 够用，Drizzle 迁移成本高且收益不明显
- **不要换数据库**：SQLite 单文件是核心优势，除非真的需要多副本
- **Passkey 是亮点**：v1.0 已完成 WebAuthn，这是竞品少有的特性
- **Bun 可推迟**：Node 22 够用，Bun 迁移可以等生态更成熟

## 四、给下一支团队的话

1. **先读完这份文档和 README**，再读 `docs/` 下 4 份文档
