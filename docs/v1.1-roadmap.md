# v1.1 技术方案文档

> 发布日期：2026-05-03  
> 状态：草稿  
> 目标版本：v1.1

## 概述

v1.1 在已有博客系统（文章、评论、标签、权限管理、密码找回、搜索）基础上新增四大功能模块：

1. 通知系统（评论回复通知）
2. 用户主页（个人资料公开页）
3. 点赞/收藏功能
4. 文件上传（头像、帖子图片）

---

## 1. 通知系统

### 1.1 技术选型

| 方案 | 延迟 | 复杂度 | 持久化 | 选型结论 |
|------|------|--------|--------|---------|
| Prisma Polling (轮询) | 高 (~5s) | 低 | 天然持久化 | 初期可用，不做优先 |
| Server-Sent Events | 低 (~实时) | 中 | 需额外存 DB | 适合但需活跃连接 |
| WebSocket (ws) | 低 (~实时) | 高 | 需额外存 DB | 当前架构过渡设计 |
| **In-App 拉取 + 红点提示** | 中 | **低** | **Prisma 原生** | **v1.1 首选** |

**选择理由：** 当前项目使用 Next.js 无状态架构，引入 WebSocket 需要额外进程维护连接。评论回复通知无需实时推送，用户下次访问或页面刷新时展示未读通知即可。采用 **通知中心模式**（类似 GitHub 通知），收益/成本比最高。

### 1.2 数据模型

```prisma
enum NotificationType {
  REPLY_TO_COMMENT     // 有人回复了你的评论
  COMMENT_ON_POST      // 有人评论了你发布的文章
  LIKE_POST            // 有人赞了你的文章
  LIKE_COMMENT         // 有人赞了你的评论
  BOOKMARK_POST        // 有人收藏了你的文章（可选，后续考虑）
}

model Notification {
  id         String           @id @default(cuid())
  type       NotificationType
  title      String           // 通知标题（简短概括）
  message    String?          // 通知正文（可选）
  link       String?          // 跳转链接（如 /blog/my-post#comment-123）
  read       Boolean          @default(false)
  createdAt  DateTime         @default(now())

  // 接收者（通知发给谁）
  recipientId String
  recipient   User   @relation(fields: [recipientId], references: [id], onDelete: Cascade)

  // 触发者（谁引发的通知）
  actorId    String?
  actor      User?   @relation(fields: [actorId], references: [id], onDelete: SetNull)

  // 关联资源（方便去重或批量处理）
  postId    String?
  post      Post?   @relation(fields: [postId], references: [id], onDelete: Cascade)

  commentId String?
  comment   Comment? @relation(fields: [commentId], references: [id], onDelete: Cascade)

  @@index([recipientId, read])
  @@index([recipientId, createdAt])
}
```

### 1.3 API 路由设计

```
通知 API — /api/notifications/*

| 方法   | 路径                       | 说明               | 角色     |
|--------|----------------------------|--------------------|----------|
| GET    | /api/notifications         | 获取通知列表       | 已登录   |
| PATCH  | /api/notifications/read    | 批量标记已读       | 已登录   |
| PATCH  | /api/notifications/[id]    | 单条标记已读       | 已登录   |
| GET    | /api/notifications/unread-count | 未读数量      | 已登录   |
```

**查询参数 (GET /api/notifications):** `page`, `pageSize`, `type`, `read`

### 1.4 触发时机

通知在 **Service 层** 触发，不侵入 Controller：

```
createComment() 成功后 →
  if (有父评论) → 通知父评论作者 (REPLY_TO_COMMENT)
  if (文章作者 ≠ 评论者) → 通知文章作者 (COMMENT_ON_POST)
```

### 1.5 组件树

```
src/
  services/notification.service.ts    ← 创建/查询通知
  app/api/notifications/
    route.ts                          ← GET / PATCH 批量
    [id]/route.ts                     ← PATCH 单条
    unread-count/route.ts             ← GET 未读数量
  components/notifications/
    NotificationBell.tsx              ← Header 右上角铃铛 + 未读红点
    NotificationPanel.tsx             ← 下拉通知面板
    NotificationItem.tsx              ← 单条通知（含类型图标）
    NotificationList.tsx              ← 通知列表（含加载更多）
  hooks/useNotifications.ts          ← SWR 封装
```

### 1.6 安全考量

- 只能查询/操作自己的通知（`recipientId` 强制等于当前用户）
- 通知创建在服务端 Service 层完成，客户端无直接创建权限
- 批量标记已读需要 `id[]` 白名单校验，防止越权标记他人通知
- 未读数量接口做缓存（SWR `dedupingInterval: 5000`），避免频繁查库

## 2. 用户主页

### 2.1 页面设计

公开页面：`/user/[username]`  
包含以下区域：

1. **个人资料卡** — 头像、用户名、bio、注册时间、徽章/角色
2. **文章列表** — 该用户已发布的文章（分页）
3. **最近评论** — 该用户最近的公开评论（分页，可选）

### 2.2 Schema 变更

User 模型已包含 `name`, `email`, `image`, `bio`, `createdAt`，无需新增字段。  
需新增 **username 唯一标识**用于 URL：

```prisma
model User {
  // ... 现有字段
  username    String?   @unique   // 新增：公开页 URL 标识
}
```

**兼容策略：** 现有用户 `username = null`，首次访问时引导设置。  
**生成规则：** 注册时自动从 email 前缀生成，允许用户后续在设置页修改。

### 2.3 API 路由

```
| 方法 | 路径                       | 说明               | 角色     |
|------|----------------------------|--------------------|----------|
| GET  | /api/users/[username]      | 获取用户公开资料   | 公开     |
| GET  | /api/users/[username]/posts| 获取用户文章列表   | 公开     |
```

**查询参数 (GET /api/users/[username]/posts):** `page`, `pageSize`

### 2.4 组件树

```
src/
  app/(public)/user/[username]/
    page.tsx                     ← 用户主页 SSR
    loading.tsx                  ← 骨架屏
    not-found.tsx                ← 用户不存在
  components/user/
    ProfileCard.tsx              ← 个人资料卡
    UserPostList.tsx             ← 用户文章列表
  hooks/useUser.ts              ← 用户数据获取
```

### 2.5 安全考量

- 公开页只暴露 `id`, `name`, `username`, `image`, `bio`, `createdAt`，不暴露 `email`, `password`
- API 层需做字段过滤，Service 层使用 `select` 限定返回字段
- `username` 修改频率限制（如 30 天内只能改一次，由前端提示 + 后端校验）
- `username` 校验规则：字母数字下划线，长度 3-20，不能纯数字

## 3. 点赞 / 收藏功能

### 3.1 数据模型

```prisma
model Like {
  id        String   @id @default(cuid())
  createdAt DateTime @default(now())

  userId    String
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  postId    String?
  post      Post?    @relation(fields: [postId], references: [id], onDelete: Cascade)

  commentId String?
  comment   Comment? @relation(fields: [commentId], references: [id], onDelete: Cascade)

  @@unique([userId, postId])        // 一篇文章只能赞一次
  @@unique([userId, commentId])     // 一条评论只能赞一次
  @@index([postId])
  @@index([commentId])
}

model Bookmark {
  id        String   @id @default(cuid())
  createdAt DateTime @default(now())

  userId    String
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  postId    String
  post      Post     @relation(fields: [postId], references: [id], onDelete: Cascade)

  @@unique([userId, postId])
  @@index([userId])
}
```

### 3.2 API 路由设计

```
点赞 API — /api/likes/*

| 方法 | 路径                          | 说明               | 角色     |
|------|-------------------------------|--------------------|----------|
| POST | /api/likes/post/[postId]      | 点赞文章           | 已登录   |
| DELETE | /api/likes/post/[postId]    | 取消点赞文章       | 已登录   |
| POST | /api/likes/comment/[commentId] | 点赞评论          | 已登录   |
| DELETE | /api/likes/comment/[commentId] | 取消点赞评论     | 已登录   |
| GET  | /api/likes/post/[postId]      | 获取文章点赞状态和数量 | 公开  |

收藏 API — /api/bookmarks/*

| 方法 | 路径                       | 说明               | 角色     |
|------|----------------------------|--------------------|----------|
| POST | /api/bookmarks/[postId]    | 收藏文章           | 已登录   |
| DELETE | /api/bookmarks/[postId]  | 取消收藏           | 已登录   |
| GET  | /api/bookmarks             | 我的收藏列表       | 已登录   |
```

**同一端点处理 toggle 的方案已被否决** — POST 为正操作、DELETE 为反操作，语义明确且便于幂等处理。

### 3.3 查询优化

文章列表/详情页需要展示点赞数和收藏数，通过 Prisma `_count` 关联查询：

```typescript
// 在 postInclude 中追加
_count: {
  select: {
    likes: true,
    bookmarks: true,
    comments: true,
  },
}
```

当前用户的点赞/收藏状态需额外查询（当前用户已登录时）：

```typescript
const userLike = await prisma.like.findUnique({
  where: { userId_postId: { userId: session.user.id, postId } },
});
```

**性能方案：** 文章列表页将用户点赞/收藏状态合并到列表查询中，避免 N+1：

```typescript
// 在 listPosts() 中，如果当前用户已登录，同时查询用户的点赞/收藏 ID 集合
const [posts, total, userLikes, userBookmarks] = await Promise.all([
  prisma.post.findMany({ ... }),
  prisma.post.count({ ... }),
  userId ? prisma.like.findMany({ where: { userId, postId: { in: postIds } }, select: { postId: true } }) : [],
  userId ? prisma.bookmark.findMany({ where: { userId, postId: { in: postIds } }, select: { postId: true } }) : [],
]);
```

### 3.4 组件树

```
src/
  services/like.service.ts         ← 点赞/取消+计数
  services/bookmark.service.ts     ← 收藏/取消
  app/api/likes/
    post/[postId]/route.ts         ← 文章点赞
    comment/[commentId]/route.ts   ← 评论点赞
  app/api/bookmarks/
    route.ts                       ← 收藏列表
    [postId]/route.ts              ← 收藏/取消
  components/interactions/
    LikeButton.tsx                 ← 点赞按钮（文章/评论通用）
    BookmarkButton.tsx             ← 收藏按钮
    BookmarkList.tsx               ← 收藏列表页组件
  hooks/useLikes.ts               ← 点赞状态管理
  hooks/useBookmarks.ts           ← 收藏状态管理
```

### 3.5 安全考量

- 点赞/收藏需要幂等校验（`@@unique` 约束保证数据库级去重）
- 取消操作需验证 `userId` 归属，防止越权取消他人点赞
- 已删除文章的点赞记录视为不存在（级联删除）
- 评论点赞同样受评论删除级联影响
- 收藏接口加频率限制（防止脚本大量操作）

## 4. 文件上传

### 4.1 技术选型

| 方案 | 存储成本 | 扩展性 | 部署复杂度 | 选型结论 |
|------|---------|--------|-----------|---------|
| 本地文件系统 | 低 | 差（多实例问题） | 低 | v1.1 首选 |
| S3/MinIO | 中 | 好 | 中 | v1.2 迁移 |
| Cloudinary/Imgur | 低（免费额度） | 好 | 低（依赖第三方） | 可做备选 |

**选择理由：** 本地部署场景优先，通过 `public/uploads/` 目录存储，Nginx 或 CDN 代理静态文件。后续可平滑迁移到 MinIO 或 S3（只需替换 Service 层实现，API 接口不变）。

**抽象策略：**

```typescript
// src/services/storage.service.ts
interface StorageService {
  upload(file: File, path: string): Promise<string>;  // 返回 URL
  delete(url: string): Promise<void>;
}

// 本地实现
class LocalStorageService implements StorageService { ... }
// 未来 S3 实现
class S3StorageService implements StorageService { ... }
```

### 4.2 上传规范

**头像：**
- 格式：JPEG, PNG, WebP
- 最大尺寸：2MB
- 分辨率限制：500x500（服务端裁剪/压缩）
- 存储路径：`/uploads/avatars/{userId}.webp`
- 推荐：使用 `sharp` 转换成 WebP 并缩放至 256x256

**帖子图片：**
- 格式：JPEG, PNG, WebP, GIF
- 最大尺寸：10MB
- 存储路径：`/uploads/posts/{postId}/{timestamp}.{ext}`
- 自动生成缩略图（640px 宽）用于文章卡片

### 4.3 API 路由设计

```
文件上传 API — /api/upload/*

| 方法 | 路径              | 说明                 | 角色     |
|------|-------------------|----------------------|----------|
| POST | /api/upload/avatar  | 上传头像           | 已登录   |
| POST | /api/upload/image   | 上传帖子图片       | 已登录   |
| DELETE | /api/upload/image | 删除已上传的图片   | ADMIN/作者 |
```

**POST /api/upload/avatar**  
Request: `multipart/form-data` with `file` field  
Response:
```json
{
  "success": true,
  "data": { "url": "/uploads/avatars/user123.webp" }
}
```

**POST /api/upload/image**  
Request: `multipart/form-data` with `file` field  
Response:
```json
{
  "success": true,
  "data": { "url": "/uploads/posts/post123/1680000000.webp" }
}
```

### 4.4 编辑器集成

TipTap 编辑器扩展 `@tiptap/extension-image` 支持图片插入。

实现自定义上传处理函数：

```typescript
// TipTap Editor 配置
Image.configure({
  allowBase64: false,  // 不允许 base64，强制走上传
  uploadFn: async (file: File) => {
    const formData = new FormData();
    formData.append("file", file);
    const res = await fetch("/api/upload/image", { method: "POST", body: formData });
    const { data } = await res.json();
    return data.url;
  },
})
```

### 4.5 组件树

```
src/
  services/storage.service.ts      ← 存储抽象层（本地实现）
  services/image.service.ts        ← 图片处理（sharp 压缩/裁剪）
  app/api/upload/
    avatar/route.ts                ← 头像上传
    image/route.ts                 ← 帖子图片上传
  components/upload/
    AvatarUpload.tsx               ← 设置页头像上传组件（含裁剪预览）
    ImageUploader.tsx              ← 编辑器图片上传组件
```

### 4.6 安全考量

- 文件类型白名单验证（MIME + magic bytes 双重校验，防止重命名绕过）
- 文件名随机化（UUID 或 timestamp），防止路径遍历
- 文件尺寸限制在 API 层和 Nginx 层双重配置
- 上传目录配置在 `.gitignore`，排除版本控制
- 图片内容安全检查（可选：接入 NSFW 检测，v1.1 暂不做）
- 头像上传自动缩放，防止大规模图片存储
- `/api/upload/image` 需要用户已登录，并且生成的文件名不可预测
- 生产环境配置反向代理（Nginx）的 `client_max_body_size` 限制

### 4.7 生产环境部署考虑

```nginx
# Nginx 配置片段
client_max_body_size 12M;

location /uploads/ {
    expires 30d;
    add_header Cache-Control "public, immutable";
    access_log off;
}
```
